Nessus to STIG: DoD Compliance Workflow

Status: In Progress – Expected completion: March 2025

RMF Phases: Assess, Authorize

Relevant Roles: ISSO, Cyber Assurance Analyst, Security Control Assessor

Project Overview

This project simulates a real-world DoD security control assessment workflow. Using a Windows VM as the target system, I run authenticated vulnerability scans with Nessus Essentials (which functions similarly to ACAS), identify findings, map them to DISA STIG IDs using STIG Viewer, create a POA&M entry, remediate the finding, and validate with a re-scan.

The goal is to demonstrate end-to-end understanding of how vulnerability management integrates with the RMF Assess and Authorize phases.

Tools Used

  • Nessus Essentials – Vulnerability scanning (ACAS equivalent)
  • STIG Viewer – DISA STIG checklist management
  • Windows 11 VM – Target system
  • VirtualBox – Hypervisor for lab environment

Step 1: Lab Environment

I set up a Windows 11 virtual machine with networking configured for host-only access. The VM is intentionally configured with common misconfigurations to generate meaningful scan findings.

Note: Screenshot of the VM setup will be added when connectivity improves.

Step 2: Initial Vulnerability Scan

I configured Nessus Essentials to run an authenticated scan against the Windows 11 VM. The scan identified several findings including:

  • SMB Signing not required (Medium severity)
  • Terminal Services Encryption Level is not FIPS-140 compliant (Medium)
  • Multiple missing Windows security patches

Note: Screenshot of scan results will be added when connectivity improves.

Step 3: STIG Mapping

Using STIG Viewer, I mapped each finding to the corresponding DISA STIG rule:

  • SMB Signing not required → WN10-SO-000075
  • Terminal Services encryption → WN10-CC-000270

This step demonstrates understanding of how scanner findings translate to specific STIG requirements—a core competency for ISSO and assessment roles.

Note: Screenshot of STIG Viewer with mapped findings will be added when connectivity improves.

Step 4: POA&M Documentation

For each finding, I created a POA&M entry with:

  • Finding description and severity
  • STIG ID reference
  • Identified root cause
  • Planned remediation steps
  • Target completion date
  • Responsible party (self, for lab purposes)

Note: POA&M template will be added as a downloadable link when connectivity improves.

Step 5: Remediation

I remediated the SMB signing finding by applying the appropriate Windows security policy: 

# PowerShell command to enable SMB signing
Set-SmbServerConfiguration -RequireSecuritySignature $true -Force

For the Terminal Services finding, I modified the Group Policy setting: Computer Configuration → Administrative Templates → Windows Components → Remote Desktop Services → Remote Desktop Session Host → Security

Step 6: Validation Scan

After remediation, I ran a new Nessus scan to verify the findings were resolved. Both SMB signing and Terminal Services findings no longer appeared in the results.

Note: Screenshot of clean validation scan will be added when connectivity improves.

Key Takeaways

  • Scanner findings (Nessus/ACAS) directly map to specific STIG controls
  • POA&M documentation must include finding details, STIG references, and remediation plans
  • Validation scanning proves fix effectiveness and supports ATO decisions
  • This workflow directly supports RMF Steps 4 (Assess) and 5 (Authorize)
