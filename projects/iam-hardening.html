AWS IAM Hardening

Completed March 2026 · AWS, IAM, CloudTrail

Executive Summary

AWS accounts start with minimal security controls. This project hardened a default AWS account to meet DoD-relevant security standards. The result is a locked-down IAM configuration with MFA enforcement, least privilege policies, and complete audit logging—ready for compliance assessments.

The Problem

The target AWS account had been used for testing and had accumulated:

  • Root account with no MFA
  • Stale IAM users with unused access keys
  • Overly permissive policies (some users had full admin)
  • No centralized audit logging
  • No visibility into who was doing what

This configuration would fail any basic compliance review and posed unnecessary security risk.

Requirements

  • Root account must have MFA enabled
  • All human access must require MFA
  • Policies must follow least privilege
  • All IAM events must be logged to CloudTrail
  • Credential report must show no stale keys
  • Must map to NIST 800-53 controls (AC-2, AC-3, AC-6, AU-2)

Approach & Alternatives Considered

I evaluated three approaches:

  • AWS IAM Access Analyzer: Good for identifying unused permissions, but doesn't enforce MFA or create policies
  • AWS Organizations SCPs: Powerful for multi-account, but overkill for single account
  • Manual policy rewrite + MFA enforcement: Most direct, gives full control, documents reasoning

I chose the manual approach because it forced me to understand every policy and decision. Automated tools can clean up, but they don't teach you why.

Implementation

1. Root Account Hardening

Enabled MFA on root account. Removed any existing access keys (there were none). Configured root email for critical alerts.

2. IAM Credential Report Analysis

Generated and reviewed the IAM Credential Report. Identified:

  • 3 users with access keys over 90 days old
  • 2 users who hadn't logged in for 6+ months
  • 1 user with no MFA enabled

Disabled stale users and keys after confirming they weren't needed.

3. Policy Rewrite

Replaced overly permissive policies with least privilege versions. Example: a developer had full S3 access but only needed read/write to two specific buckets.

Before:

{
  "Effect": "Allow",
  "Action": "s3:*",
  "Resource": "*"
}

After:

{
  "Effect": "Allow",
  "Action": [
    "s3:ListBucket",
    "s3:GetObject",
    "s3:PutObject"
  ],
  "Resource": [
    "arn:aws:s3:::dev-bucket",
    "arn:aws:s3:::dev-bucket/*",
    "arn:aws:s3:::prod-bucket",
    "arn:aws:s3:::prod-bucket/*"
  ]
}

4. MFA Enforcement

Created an MFA enforcement policy and attached it to all human users:

{
  "Effect": "Deny",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "BoolIfExists": {
      "aws:MultiFactorAuthPresent": "false"
    }
  }
}

5. CloudTrail Configuration

Enabled CloudTrail across all regions with log file validation. Configured logs to be stored in a secure S3 bucket with access logging enabled.

Trade-offs & Decisions

  • MFA enforcement on all API calls: This can break automation. I documented which users were human vs. service accounts and excluded service accounts from the MFA policy.
  • Manual vs automated remediation: Manual took longer but ensured I understood every change. Future iterations could use Terraform.
  • Granularity of policies: Could have made policies even more granular (down to specific API actions), but balanced security with maintainability.

Results

  • ✅ Root account MFA enabled
  • ✅ All human users require MFA
  • ✅ 4 stale users removed, 3 old access keys disabled
  • ✅ Every IAM action logged to CloudTrail
  • ✅ All policies follow least privilege

RMF Control Mapping

Control Implementation
AC-2 (Account Management) IAM user lifecycle, credential rotation via Credential Report, stale account removal
AC-3 (Access Enforcement) IAM policies enforce authorized access
AC-6 (Least Privilege) All policies scoped to minimum required permissions
AU-2 (Audit Events) CloudTrail logs all IAM events

What I Learned

  • Least privilege takes time. You can't do it in one pass. Start broad, then refine as you understand usage patterns.
  • MFA enforcement needs exceptions. Service accounts can't use MFA. Document them clearly.
  • Credential Report is underrated. It's a free, simple way to audit your IAM hygiene.
  • CloudTrail is essential. Without logs, you have no idea what's happening.
