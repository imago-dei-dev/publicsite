Executive Summary

This RMF package was developed for a system that provides real-time tracking and documentation of high-value assets using AWS cloud services. The system handles sensitive data at a moderate impact level. Five NIST 800-53 controls were selected based on what the system owner could actually implement and test. Two high findings, two medium findings, and one low finding were documented in the POA&M with remediation plans and target dates.

Control Selection Rationale

I selected five controls because they represented the security functions the system owner could actually implement and test. Physical controls are inherited from AWS. Backup controls are handled by the infrastructure team. These five controls sit at the system boundary and directly protect sensitive data.

AC-3 (Access Enforcement)

This was my first priority. The system handles sensitive data, and without access controls, none of the other controls matter. IAM enforces least privilege for all users and processes. Security groups restrict network access to authorized IP ranges. API Gateway controls external access with API keys.

SC-7 (Boundary Protection)

External partners access the system through API endpoints. That is the primary attack surface. I needed to show that traffic is monitored and controlled at the boundary before anything reaches internal systems. The VPC is segmented into public and private subnets, with application servers in private subnets that have no direct internet access.

AU-2 (Event Logging)

Every compliance framework requires audit trails. I selected this early because logs are required for monitoring and incident investigation. CloudTrail monitors and logs system actions. Logs are ingested by CloudWatch and stored in S3 for long-term retention.

SI-4 (System Monitoring)

Logging without monitoring is useless. I wanted to show continuous visibility, not just point-in-time compliance. GuardDuty analyzes CloudTrail and VPC Flow Logs to detect indicators of compromise. CloudWatch triggers alerts when threats are detected.

IA-2 (Identification and Authentication)

MFA and unique credentials are non-negotiable for a system handling sensitive data. I prioritized this because shared accounts would break all other controls. IAM Identity Center centralizes authentication across the organization. MFA is enforced for all privileged accounts.

Control Implementation (AWS)

AC-3 (Access Enforcement)

AWS IAM enforces access control policies for all internal users and processes. IAM roles with least privilege permissions are assigned to EC2 instances and RDS databases. Security groups restrict network access to authorized IP ranges. API Gateway controls external access using API keys, with CloudTrail logging all access attempts.

SC-7 (Boundary Protection)

API Gateway acts as the external boundary, authenticating all incoming requests before they reach the VPC. Inside the VPC, security groups control traffic to EC2 instances and RDS databases. The VPC is segmented into public and private subnets, with application servers in private subnets that have no direct internet access.

AU-2 (Event Logging)

CloudTrail monitors and logs system actions. Logs are ingested and analyzed by CloudWatch. EventBridge uses data from CloudWatch Logs and Insights to send SNS messages for specified activity. S3 buckets are used for long-term storage.

SI-4 (System Monitoring)

CloudTrail records all API calls. VPC Flow Logs capture network traffic data. GuardDuty analyzes both to detect indicators of compromise and misuse. CloudWatch monitors GuardDuty findings and triggers SNS alerts for security personnel when threats are detected.

IA-2 (Identification and Authentication)

IAM Identity Center centralizes authentication across the organization's multiple accounts and applications. Identity-based policies enforce least privilege for all IAM users and roles. Resource-based policies control access to S3 buckets containing audit logs. MFA is enforced for all privileged accounts via IAM policy. CloudTrail logs all authentication attempts.

Risk Register Highlights

R-004: Stale Credentials (High)

Likelihood: 5 | Impact: 4 | Score: 20

Credentials of terminated employees and contractors were not being removed from the system. I scored this as High because one inactive admin key could expose all sensitive data. The remediation plan is to run an IAM credential report, remove stale credentials, implement quarterly audits, and include credential removal in the employee offboarding process.

R-002: S3 Bucket Misconfiguration (Medium)

Likelihood: 3 | Impact: 4 | Score: 12

Improperly configured S3 bucket policies could allow unauthorized users to access logs of other customers' data. The remediation plan is to implement bucket policies restricting access to database management staff and enforce Block Public Access at the account level.

R-006: Mobile Device Data Exposure (Medium)

Likelihood: 5 | Impact: 3 | Score: 15

Customers' employees using the mobile app could lose or mishandle devices containing cached sensitive data. I transferred this risk to the customer via Terms of Service. The system cannot enforce remote wipe or device monitoring, so the customer must accept that responsibility.

R-008: CUI Misclassification (Medium)

Likelihood: 4 | Impact: 4 | Score: 16

Prime contractors do not clearly communicate to subcontractors which elements are designated as sensitive data. I accepted this risk because the system has no authority over prime contractors. I documented the acceptance and set a reassessment date.

R-010: Vendor Non-Compliance (Medium)

Likelihood: 5 | Impact: 3 | Score: 15

A subcontractor handling sensitive logistics data runs their operations on the clear web using unencrypted drives and HTTP. I transferred this risk to procurement. The contract requires FedRAMP Moderate compliance. If the vendor does not comply, we replace them.

What I Learned

This project taught me that compliance is not about checking boxes. It is about understanding what the system actually does, who owns each control, and where the real risk lies. I learned to push back on controls that looked good on paper but could not be implemented or tested. I also learned to document risk acceptance decisions clearly so that leadership understands what they are signing off on.

If I were to do this again, I would add a baseline configuration control (CM-2) to track AMI versions. We had no visibility into EC2 patching, and that is a gap I would close in a future package. I would also escalate any risk with a score over 18 directly to the CISO rather than waiting for the next review cycle.
